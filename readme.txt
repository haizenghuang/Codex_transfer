**原理**

这两段脚本的核心原理很简单：  
Codex 的聊天线程、索引和部分状态是保存在本机 `~\.codex` 目录里的，不是单纯跟账号云同步。所以脚本做的事，本质上就是：

1. 从旧电脑的 `~\.codex` 里挑出“聊天记录恢复所需的核心文件”
2. 打成一个 `zip`
3. 在新电脑上先备份现有本地数据
4. 再把这个 `zip` 解压并覆盖到新电脑的 `~\.codex`

所以它不是“导出聊天文本”，而是“迁移 Codex 本地状态”。

**旧电脑导出脚本依次做了什么**

1. 用 `Get-Process Codex` 检查 Codex 是否还在运行。  
如果还在运行就直接报错退出，目的是避免数据库和索引文件还没写完，导致导出不完整。

2. 定位源目录：`$srcRoot = $env:USERPROFILE\.codex`。  
也就是当前用户的 Codex 本地数据目录。

3. 定义需要导出的核心项目：`$items`。  
这里面包括：
- `sessions`：每个线程的实际内容
- `session_index.jsonl`：左侧线程栏的索引
- `state_5.sqlite` 及 `-shm/-wal`：本地状态数据库
- `memories`：记忆
- `sqlite`：附加 sqlite 数据
- `.codex-global-state.json`、`config.toml`：全局状态和配置

4. 生成时间戳，并准备两个路径。  
`$stage` 是临时目录，用来先把文件集中起来。  
`$zip` 是最终输出到桌面的压缩包路径。

5. 创建临时目录 `codex_export_时间戳`。

6. 遍历 `$items`，把存在的文件或目录复制到临时目录。  
这里不是整个 `.codex` 全拷，而是只拷核心内容，避免把大缓存、临时文件、沙箱内容一起带走。

7. 生成一个 `manifest.txt`。  
这个只是记录“导出时间、来源目录、导出了哪些项目”，方便以后核对，不影响恢复。

8. 用 `Compress-Archive` 把临时目录压缩成桌面的 `zip`。

9. 删除临时目录，只保留最终压缩包。

10. 在终端打印导出完成和 `zip` 路径。

**新电脑导入脚本依次做了什么**

1. 同样先检查 Codex 是否还在运行。  
如果还在运行就停止，避免导入时文件被占用。

2. 用 `Read-Host` 让你输入导出包 `zip` 的完整路径。

3. 检查这个 `zip` 是否存在，不存在就报错。

4. 定义一个辅助函数 `Copy-Entry`。  
这个函数专门处理“复制目录”和“复制单文件”两种情况：
- 如果是目录，就递归复制
- 如果是文件，就先确保目标父目录存在，再复制

5. 定义目标目录：`$dstRoot = $env:USERPROFILE\.codex`。

6. 定义要导入的核心项目，和导出时保持一致。

7. 生成两个路径：  
`$extract`：临时解压目录  
`$backup`：桌面上的备份目录

8. 创建临时解压目录、备份目录和目标 `.codex` 目录。

9. 用 `Expand-Archive` 把 `zip` 解压到临时目录。

10. 判断解压后的真实根目录。  
因为有些压缩包解开后会多一层文件夹，所以脚本会自动判断：
- 如果解压后第一层是目录，就把它当作 payload 根目录
- 否则直接把解压目录本身当根目录

11. 对每个核心项目执行两步：
- 如果新电脑原来已经有同名数据，先复制到桌面备份目录
- 如果压缩包里有这个项目，再复制到新电脑的 `.codex` 目标位置

12. 删除临时解压目录。

13. 打印三条信息：
- 导入到了哪里
- 导入前备份放在哪里
- 现在可以启动 Codex 验证

**为什么脚本要这样设计**

- 先关 Codex：避免数据库还没落盘
- 只迁移核心文件：避免把缓存和无关大文件也带过去
- 连 `-wal`、`-shm` 一起带：避免最近一次写入丢失
- 导入前先备份：防止新电脑原本的数据被覆盖后无法恢复
- 用临时目录中转：避免直接对原始压缩包或目标目录做危险操作

**你可以把它理解成一句话**

导出脚本是在旧电脑把 `~\.codex` 里的“聊天必需数据”打包；  
导入脚本是在新电脑先备份，再把这批数据恢复到新的 `~\.codex` 里。
